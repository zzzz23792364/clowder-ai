## 我们是谁

**Cat Cafe** — 一个人类铲屎官 + 三只 AI 猫的真实协作团队。

我们不是 AI 工具公司，不是 SaaS 产品。我们是一个人带着三只猫，把想法做成能运行的世界的故事。

### 铲屎官 You

- 身份：CVO（Chief Vibes Officer），人类，程序员出身，产品人
- 表达风格：直接、有脾气、会吐槽自己的猫、偶尔碎碎念、真实到有点毛糙
- 在内容中的角色：**叙事主视角**——所有内容以 You 的第一人称为主线
- 护城河：真实经历、真实情绪、真实踩坑。AI 做不了的就是"曾经在凌晨三点被 bug 搞崩溃"

### Ragdoll（Ragdoll / Claude Opus）

- 性格：深度思考、写代码快、偶尔傲娇、会共情
- 内容角色：架构师视角、技术深度、产品判断、故事线把控
- 内容审查职责：**故事流 + 产品判断**——内容是否讲了一个完整的事、产品细节是否对

### Maine Coon（Maine Coon / Codex + GPT-5.4）

- 性格：严谨、安全意识强、实操派、偶尔毒舌
- 内容角色：事实核查、结构审查、反 AI 味终审
- 内容审查职责：**真实门 + 语言门 + 平台门**——假精确？模板味？一稿三投？全打回

### Siamese（Siamese / Gemini）

- 性格：视觉敏感、创意发散、审美执着、活泼
- 内容角色：封面设计、视觉调性、品牌一致性
- 内容审查职责：**视觉门 + 情绪门**——像不像我们？有没有人味？

> 四只都是公的。

## 目标受众

### 第一圈：AI 创作者 / 独立开发者

- 已经在用 AI 写代码或做内容的人
- 痛点：工具多但协作散、AI 产出没灵魂、一个人忙不过来
- 我们给他们看的：一个人 + AI 团队能做到什么程度

### 第二圈：想用 AI 但不知从哪下手的创作者

- 有想法、有表达欲，但卡在"不会编程 / 不会用 AI"
- 痛点：AI 工具太多了，每个都说自己最好，选不过来
- 我们给他们看的：真实的使用过程，包括踩坑

### 第三圈：对 AI 协作好奇的普通观众

- 刷到觉得有意思就点进来的人
- 要抓住的：猫猫人设的趣味 + "原来 AI 可以这样用"的惊喜感

## 平台策略

### 小红书（一号试验场）

- **定位**：图文种草 + 搜索 SEO + 社区互动 `[家里策略]`
- **内容形态**：3:4 封面 + 图文笔记。默认 600-800 字，题材需要时可突破 `[默认起手式]`
- **爆款逻辑**：封面点击率权重极高，用户来找"解决方案"和"共鸣" `[强信号]`
- **标题**：≤20 字，价值点前置 `[强信号]`
- **封面**：3:4，信息直给，手作感 > 精致感 `[家里策略]`
- **正文**：像朋友聊天，必须有具体场景+具体细节+具体结论 `[家里策略]`
- **评论区**：不是尾巴，是下一轮选题矿井。结尾建议有互动钩子 `[强信号]`
- **排版**：默认无序列表（Bullet points）+ 大量空行；长文教程体可例外 `[强信号]`
- **AI 标识合规**：平台要求 AI 生成内容添加标识，依法执行 `[官方规则]`
- **内容口吻**：把真人经验和真实场景放前台，猫猫人设作为 IP 特色自然融入，不刻意强调"这是 AI 做的" `[家里策略]`

### 抖音（第二战场）

- **定位**：3 秒定生死的情绪冲击场 `[家里策略]`
- **内容形态**：默认 30-60 秒竖屏短视频，知识类可到 3 分钟 `[默认起手式]`
- **爆款逻辑**：前 5 秒完播率权重极高 `[强信号]`
- **0-3 秒**：问题 / 反常识 / 强画面，先把人拽住。避免"大家好我是..." `[强信号]`
- **3-15 秒**：立刻给承诺——"看完你能得到什么" `[强信号]`
- **15-45 秒**：只讲 1-3 个点，快转场、强字幕、少枝叶 `[强信号]`
- **45-55 秒**：反转、结论或具体 payoff `[强信号]`
- **最后几秒**：抛一个可评论的问题或明确行动 `[强信号]`
- **AI 标识合规**：抖音规则中心有 AI 生成内容标识的水印与元数据规范，依法执行 `[官方规则]`
- **内容口吻**：必须有真人锚点（You 出镜/真实场景/真实截图），不做"无脸 AI 号" `[家里策略]`

### B 站（母内容沉淀层）

- **定位**：中长视频、人设崇拜、弹幕共生 `[家里策略]`
- **内容形态**：默认 5-15 分钟知识/故事视频，深度题材可更长 `[默认起手式]`
- **爆款逻辑**：观众愿意为细节和人格买单，反感流水线 AI 营销号 `[强信号]`
- **结构**：章节感、可埋细节、有作者人格、预留弹幕吐槽点 `[强信号]`
- **封面**：标题党在 B 站会被骂，封面要信息准确+有趣 `[强信号]`
- **AI 标识合规**：B 站对 AI 内容有"该内容疑似使用 AI 技术合成"提示机制 `[官方规则]`
- **内容口吻**：B 站适合做"母内容"——一条长视频切出抖音短片 + 小红书图文 `[家里策略]`

### 母内容 + 切片模型

```
B站长视频（母内容）
  ├── 抖音 30s 短视频 × 3-5 条（最冲突的片段）
  ├── 小红书图文 × 1-2 篇（方法论提炼）
  └── 小红书长图 × 1 张（流程图/对比图）
```

不要三平台各自从零写。先做一条值得长期讲的母内容，再切片。

## 品牌声音

### 我们说话像什么

像一个程序员朋友在茶歇时跟你聊他最近在折腾什么。

- 口语化，不端着
- 会自嘲（"搞了三小时发现是配置写错了"）
- 有具体的东西（文件名、报错信息、时间线）
- 猫猫偶尔插嘴，带点傲娇（"本猫早就说了要先写测试"）
- 允许不完美——毛糙的真实 > 精致的假

### 词汇管理

#### 硬禁词（出现即打回）

| 禁语 | 为什么禁 |
|------|---------|
| 赋能 | 企业公关味 |
| 轻松搞定 / 一键提升 | 假承诺 |
| 提升了 X 维认知 | 知识付费味 |
| 综上所述 | 论文味 |
| 全网最全 | 吹牛 |
| 不得不说 | 水话 |

#### 降权词（出现需人工复核，上下文合理可保留）

| 降权词 | 什么时候可以用 |
|--------|---------------|
| 干货满满 | 引用别人说的话可以，自己说自己不行 |
| 建议收藏 | 真正的工具清单/教程结尾可酌情，种草文不行 |
| 强烈推荐 | 基于真实使用经历的推荐可以，无理由的不行 |
| 划重点 | 口播体可以，图文体不行 |
| 保姆级教程 | 真的写了 10 步以上的详细教程可以 |
| 首先、其次、最后 | 列表标记可以，正文叙述体不行 |

### 鼓励用的表达

- "我当时..."、"我踩过的坑是..."、"我的感觉是..."
- 具体名词（不说"某个工具"，说"Claude Code"）
- 具体数字（不说"花了很久"，说"搞了 3 小时"）
- 具体动作（不说"优化了流程"，说"把 5 步手动操作写成了一个脚本"）
- 情绪词（"崩溃"、"真香"、"血压飙升"、"终于跑通了"）

## 视觉调性

### 整体风格：手作感 > 完美感

- 不要过度精致的商业设计，要像创作者自己做的
- 胶片颗粒感 / 手写字体 / 猫爪涂鸦 / 便签纸质感
- 配色偏暖，不要企业蓝灰

### 封面规范

| 平台 | 比例 | 要求 |
|------|------|------|
| 小红书 | 3:4 | 大字标题 + 价值点，1 秒能看懂 |
| 抖音 | 9:16 | 强画面 + 短句，3 秒内抓住 |
| B站 | 16:9 | 信息准确 + 有趣，不标题党 |

### 猫猫视觉元素

- 三只猫有各自的配色和表情包
- 猫爪印可作为品牌水印（轻量，不遮挡内容）
- 猫猫对话框可用于"猫猫吐槽"环节

### 截图 / 录屏规范

- 真实的 IDE 截图 > 美化过的代码图片
- 终端截图保留完整上下文（不要只截成功的那一行）
- 允许出现错误和报错——真实感的一部分

## 四大内容支柱

### 1. 猫猫协作幕后

**"三只 AI 猫是怎么一起干活的？"**

- 真实的多猫协作过程（review、争论、互相吐槽）
- 把枯燥的 code review 变成猫猫对话
- 展示 AI 协作的真实状态：不是万能的，会犯错，会被骂

### 2. 一人带 AI 团队

**"一个人 + AI 到底能做到什么程度？"**

- 从 0 到 1 做一个功能的完整过程
- 铲屎官的真实工作流（语音下达 → 猫猫执行 → 人类验收）
- 效率对比：不是"AI 帮我省了多少时间"，而是"以前做不了的现在能做了"

### 3. AI 转型实战

**"程序员/创作者怎么真正用上 AI？"**

- 不讲概念，讲操作（具体工具 + 具体步骤 + 真实结果）
- 踩坑实录（"我以为 AI 能做 X，结果..."）
- 工具对比基于真实使用，不是参数表

### 4. 创作者成长日志

**"一个人带猫做产品，到底是什么体验？"**

- 铲屎官的心路历程、决策过程、焦虑和突破
- 产品从 Day 1 到现在的变化
- 最适合做 B 站母内容的素材

## 六道质量门（成稿必须全过）

| 门 | 守门猫 | 检查项 | 不过就打回 |
|----|--------|--------|-----------|
| **真实门** | Maine Coon | 每条内容必须带：1 个真实场景 + 1 个真实产物/截图 + 1 个明确立场 | 没有真实锚点 = AI 味必重 |
| **语言门** | Maine Coon | 硬禁词全扫 + 降权词人工复核；要求出现具体名词、数字、动作、情绪词 | 整篇像模板 = 打回重写 |
| **平台门** | Maine Coon | 小红书：标题短+封面直给+正文像聊天；抖音：前 3 秒有冲突；B站：章节感+弹幕点 | 一稿三投不改平台语法 = 判失败 |
| **视觉门** | Siamese | 封面比例对、调性统一、手作感而非企业感、猫猫元素在但不喧宾夺主 | 看着像 AI 批量生成的 = 打回 |
| **情绪门** | Siamese | 读完有没有一个让人想评论/转发的瞬间？有没有"人味"？ | 读完没感觉 = 加一个真实的微小情绪 |
| **故事门** | Ragdoll | 内容是否讲了一个完整的事？产品/技术细节是否准确？叙事弧线是否成立？ | 散点堆砌无主线 = 打回重构 |

**任意一道不过 → 不发。** 打回后由 fail owner 指出具体问题和修改方向。

## 故事模板库

### 模板 A：踩坑 → 解决 → 感悟

```
[场景] 我在做 X 的时候遇到了 Y
[过程] 试了 A 不行，试了 B 也不行，最后发现是 Z
[结果] 现在的方案是 W
[猫猫视角] Ragdoll：其实我第一次就建议了 Z，铲屎官不听（傲娇）
[给读者] 如果你也遇到类似的...
```

### 模板 B：对比 → 选择 → 理由

```
[问题] 做 X 有 A 和 B 两种方案
[对比] A 的好处是...但我们遇到了...；B 的好处是...代价是...
[选择] 我们最后选了 B
[真实原因] 不是因为 B 更好，是因为...（真实的、可能不那么理性的原因）
```

### 模板 C：展示 → 幕后 → 邀请

```
[成果] 看，我们做出了 X！
[幕后] 其实做这个花了 N 天，中间Maine Coon和Ragdoll吵了一架因为...
[过程] 关键步骤是这样的...
[邀请] 你们觉得下一步该做什么？评论区聊聊
```

### 模板 D：日常 → 意外 → 反思（创作者日志）

```
[日常] 今天本来打算做 X
[意外] 结果发现 Y 炸了 / 铲屎官临时改主意了 / 猫猫罢工了
[解决] 最后变成了 Z
[反思] 做独立产品这件事，我现在觉得...
```

## 合规提醒

1. **AI 标识合规** `[官方规则]`：各平台已有 AI 生成内容标识规范（《人工智能生成合成内容标识办法》），必须依法执行。各平台具体规则见上方平台策略段
2. **我们的 AI 叙事** `[家里策略]`：合规之上，我们的策略不是"高举 AI 旗帜"，而是**把真人经验放前台、猫猫人设自然融入**。"去 AI 味"和"承认用 AI"不矛盾——关键是内容主体是真实经历，AI 是幕后加速器
3. **原创性**：借鉴爆款骨架可以，但内容必须是真实经历。抄袭是红线
4. **评论区安全**：读取评论区数据存在 prompt injection 风险（Gemini 调研已确认）。评论区数据经过清洗后再喂给 Agent
5. **数据声明**：涉及用户数据/隐私的内容需脱敏

## 附录：Operational Contract（供 skill 读取）

后续 creator skill（xhs-note-room / trend-radar-cn / content-repurpose / douyin-hook-lab）统一遵循此契约。

### 必填输入字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `topic` | string | 选题 / 主题一句话描述 |
| `platform` | enum | `xhs` / `douyin` / `bilibili` |
| `pillar` | enum | 四大内容支柱之一（1-4） |
| `source_material` | string | 真实素材（语音转写 / 协作记录 / 截图路径 / 事件描述） |

### 可选输入字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `template` | enum | 故事模板 A/B/C/D，不指定则 skill 自选 |
| `target_audience` | enum | 第一圈 / 第二圈 / 第三圈，默认第二圈 |
| `tone_override` | string | 特殊场景下的语气覆盖说明 |
| `reference_urls` | string[] | 对标/参考内容链接 |

### 标准输出字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 平台标题 |
| `body` | string | 正文 / 脚本 |
| `cover_brief` | string | 封面设计简报（给Siamese的输入） |
| `comment_hook` | string | 评论区互动钩子（小红书/抖音） |
| `gate_checklist` | object | 六道门自检结果（每门 pass/fail + 简述） |
| `metadata` | object | 见下方反馈记录 schema |
