# Research Prompt 骨架模板

> 版本：v1.0 | 来源：Harness Engineering Round 3 三猫共识（2026-04-15）
> 设计哲学：Research prompt 本身就是 harness engineering——编码我们的决策边界给 research agent，让输出过拟合到我们的判断框架。

## 使用方法

1. 复制下方模板
2. 填充 8 个槽位（标 `{必填}` 的不能留空，标 `{推荐}` 的可酌情省略）
3. 发给云端 research agent（GPT Pro Deep Research / Gemini Deep Think）
4. 收到输出后，对照 Output Schema 检查质量

## 模板正文

```markdown
# Research Brief: {标题} {必填}

## 1. Problem Frame（任务边界）{必填}

**我们要回答的问题**：
{用 1-3 句话描述核心问题}

**非目标（明确排除）**：
- {列出不在本次调研范围内的相关话题}

**为什么现在要研究这个**：
{驱动力：哪个 Feature/ADR/讨论催生了这个研究需求？}

## 2. Current Hypotheses（我们的假设）{必填}

我们目前的判断是：
1. {假设 1}
2. {假设 2}
3. ...

**证据缺口**：
- {我们缺什么才能确认或推翻这些假设？}

> ⚠️ 这些是我们的初步假设，不是结论。请在调研中验证或推翻它们。

## 3. Disconfirm First（先找反例）{必填}

在给出支持性证据之前，请优先：
1. 寻找我们假设可能错误的证据
2. 寻找采用了相反方案但成功的案例
3. 识别我们可能存在的盲区或认知偏差

## 4. Source Mix Quota（来源配额）{推荐}

请确保来源覆盖以下类型（不必每类都有，但不能只有一种）：
- [ ] 学术论文 / 正式研究报告
- [ ] 工程博客 / 技术复盘（优先一手工程实录，非 AI 生成总结）
- [ ] 开源项目实现（GitHub repo、design doc）
- [ ] 竞品/同行方案文档

## 5. Local Constraints（我们的约束）{必填}

调研结论必须在以下约束下可行：
- **多引擎协作**：我们是多 AI 引擎（Claude/GPT/Gemini）共同工作，不是单 agent
- **人在环路**：关键决策由人类 CVO 拍板，猫建议但不自主执行高风险操作
- **知识在 repo 里**：知识沉淀在 git 仓库的 markdown 文件中，不是外部数据库
- {补充本次调研的特定约束}

## 6. Output Schema（输出格式）{必填}

请按以下结构组织输出：

### 支持我们假设的证据
| 证据 | 来源 | 置信度（高/中/低） | 可验证性 |
|------|------|---------|---------|

### 反对我们假设的证据
| 证据 | 来源 | 置信度（高/中/低） | 影响评估 |
|------|------|---------|---------|

### 我们没考虑到的维度
| 维度 | 为什么重要 | 建议的调研深入方向 |
|------|----------|------------------|

### 置信度总评
- 假设 1：{支持/反对/未定} — 理由
- 假设 2：{支持/反对/未定} — 理由

## 7. Decision Interface（决策映射）{必填}

对于每个调研发现，请标注建议的行动：
- **采纳**：证据充分，建议我们直接采用
- **试点**：有潜力但需要小范围验证
- **搁置**：当前不适用或证据不足

并说明如何落地到我们现有的 Feature / ADR / 工程体系中。

## 8. Risk Register（风险登记）{推荐}

如果我们基于本次调研结论做决策，最可能出错的地方是：
1. {风险 1}：{为什么可能错} → {缓解建议}
2. {风险 2}：{为什么可能错} → {缓解建议}
```

## 自动注入清单（发送前检查）

发送 research prompt 前，发起猫应自动注入以下本地锚点：

- [ ] **当前 Feature spec**：附上相关 Feature 的 Why/What 节摘要
- [ ] **相关 ADR**：附上最相关的 1-2 个 ADR 的核心决策
- [ ] **最近教训**：附上最相关的 1-2 条 LL，让 agent 知道我们踩过什么坑
- [ ] **BACKLOG 上下文**：当前 Feature 的前驱/后继关系

这些锚点让 research agent 知道"我们走到了哪一步"，避免给扫盲式废话。

## Quality Gate（回收检查）

收到 research 输出后，按以下标准判断质量：

| 检查项 | 通过标准 |
|--------|---------|
| 反例覆盖 | 至少有 1 条反对证据（全部支持 = 可能确认偏误） |
| 来源多样性 | 至少 2 种不同类型的来源 |
| 本地约束对齐 | 结论逐条对齐了 §5 的约束 |
| 可行动 | 每个发现有明确的 采纳/试点/搁置 标注 |
| 风险意识 | 有至少 1 条"如果我们错了"的分析 |

未通过 Quality Gate 的输出应要求 agent 补充，或降级为"参考但不作为决策依据"。
